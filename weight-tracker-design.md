# 体重管理PWA 設計書(MacroFactor代替)

## 1. 概要

### 1.1 目的
MacroFactorのアダプティブTDEEアルゴリズムを再現し、あすけんの記録データと日々の体重から、摂取カロリー目標を週次で自動調整する個人用Webアプリを構築する。

### 1.2 基本コンセプト
- **消費カロリーは直接測定しない**。摂取カロリー実績と体重トレンドの差分からTDEE(総消費カロリー)を逆算する(MacroFactor方式)
- あすけんは食事記録ツールとして継続利用。本アプリは「日次合計の転記先 + アルゴリズムエンジン + ダッシュボード」
- 入力は1日2回・計30秒以内(朝:体重、夜:あすけん日次合計)

### 1.3 ユーザー前提
- 単独利用、iPhone Safari(ホーム画面追加のPWA)
- 身長171cm、開始体重約79kg、目標73kg
- 目標ペース:週0.43%(週あたり約-0.3kg)
- タンパク質目標:1.7g/kg(トレンド体重基準)

### 1.4 あすけん無料版の制約と対応
あすけん無料版は各食事の食品名・kcalは表示されるが、P/F/Cを含む日次栄養素の「摂取量」列はロックされ数値が見えない(グラフバーのみ表示、数値は非表示)。

**対応方針**:あすけんのスクショ(各食事欄:食品名+kcal一覧)をClaudeチャットに渡し、Claudeが食品名からP/F/Cを推定・合計する運用とする。PWA自体には画像解析機能を持たせず、あくまで「数値入力→アルゴリズム→ダッシュボード」に専念させ、単一HTML・サーバー不要という設計方針を維持する。

## 2. アーキテクチャ

### 2.1 構成
```
GitHub Pages(静的ホスティング)
└── index.html(単一ファイル)
    ├── HTML(UI構造)
    ├── CSS(インライン、iPhone Safari最適化)
    ├── JavaScript(バニラJS、フレームワーク不使用)
    ├── manifest(インラインJSON、PWA化)
    └── Service Worker(オフライン対応、任意)

データ永続化: localStorage
バックアップ: JSONエクスポート/インポート機能
```

### 2.2 技術選定理由
| 項目 | 選定 | 理由 |
|------|------|------|
| ホスティング | GitHub Pages | 無料・既存バックギャモンPWAと同一運用 |
| フレームワーク | なし(バニラJS) | 単一HTML方針、依存ゼロ、Claude Codeで保守容易 |
| チャート | SVG手描き or Canvas | CDN依存を避ける。折れ線2本(実測・トレンド)程度なので自作で十分 |
| ストレージ | localStorage | 数年分でも数百KB程度。容量問題なし |
| ヘルスケア連携 | v1では非対応 | Web PWAからHealthKitへの直接アクセスは不可。v2でiOSショートカット連携を検討 |
| 栄養素解析 | Claudeチャット(手動連携) | あすけん無料版はP/F/C数値が非表示のため、スクショをClaudeに渡し食品名から推定・合計する運用とする(1.4節参照) |

### 2.3 非機能要件
- オフライン動作(Service Worker、キャッシュファースト)
- localStorage破損・消去に備え、JSONエクスポートを1タップで実行可能に
- 画面はiPhone縦持ち専用(375〜430px幅想定)

## 3. データモデル

### 3.0 日次データ入力の運用フロー
```
① 食事のたび、あすけんに記録(通常運用)
② 1日の終わりに、朝食・昼食・夕食・間食の該当スクショを
   Claudeチャット(本チャット)に渡す
③ Claudeが食品名からP/F/C・合計kcalを推定して回答
④ 算出結果をPWAの「摂取記録」欄に手入力(kcal/P/F/C 4項目・10秒)
⑤ PWAはTDEE逆算・週次目標見直しを実行
```
- **推定誤差への耐性**:食品名ベースの概算はあすけん実測値と多少ズレるが、TDEE逆算は「摂取実績と体重トレンドの差分」で補正がかかる方式のため、多少のP/F/C推定誤差は体重変化で自己修正される。数値の完璧さより記録の継続性を優先する
- **将来の自動化余地**:Claude APIの画像入力機能を使えばPWA内で画像アップロード→自動算出も技術的には可能(Phase 4候補)。ただし精度・実装コストの観点からv1では見送り、チャット経由の手動連携とする

### 3.1 localStorage スキーマ
```javascript
// key: "wt_config"
{
  "heightCm": 171,
  "goalWeightKg": 73.0,
  "weeklyLossRatePct": 0.43,      // 週あたり体重減少率(%)
  "proteinPerKg": 1.7,            // g/kg(トレンド体重基準)
  "fatRatio": 0.30,               // 残カロリーのうち脂質に配分する割合(バランス型)
  "kcalPerKgFat": 7200,           // 体脂肪1kgあたりのkcal換算
  "dayMultipliers": {             // 曜日別カロリー配分(週平均=1.0)
    "mon": 0.95, "tue": 0.95, "wed": 0.95, "thu": 0.95,
    "fri": 0.95, "sat": 1.113, "sun": 1.113
  },
  "emaAlpha": 0.1,                // 体重トレンドEMA平滑係数
  "tdeeWindowDays": 21,           // TDEE推定に使う日数
  "initialTdee": 2324             // 初期TDEE(MacroFactorの推定値を引き継ぎ)
}

// key: "wt_entries"(日次レコードの配列、date昇順)
[
  {
    "date": "2026-07-19",        // ISO 8601、ローカル日付
    "weightKg": 78.6,            // 朝の体重(null可: 未計測日)
    "intakeKcal": 1850,          // あすけん日次合計(null可: 未入力日)
    "proteinG": 130,
    "fatG": 58,
    "carbG": 195,
    "note": ""                   // 任意メモ(外食・チートデイ等)
  }
]

// key: "wt_weekly"(週次計算結果のキャッシュ、月曜起点)
[
  {
    "weekStart": "2026-07-20",
    "estimatedTdee": 2280,
    "targetAvgKcal": 1905,
    "targetByDay": { "mon": 1810, ..., "sat": 2120, "sun": 2120 },
    "trendWeightKg": 78.4,
    "targetMacros": { "proteinG": 133, "fatG": 63, "carbG": 200 }
  }
]
```

### 3.2 データ検証ルール
- weightKg: 40〜150の範囲外は入力エラー
- intakeKcal: 500〜5000の範囲外は確認ダイアログ
- 同一日付の重複登録は上書き(確認あり)

## 4. コアアルゴリズム

### 4.1 体重トレンド(EMA)
日々の水分・グリコーゲン変動をならすため、指数移動平均を使用。

```
trend[0] = weight[0]
trend[i] = trend[i-1] + α × (weight[i] − trend[i-1])   // α = 0.1
```

- 体重未計測日はEMA更新をスキップ(前回trendを保持)
- UIには「実測値(薄い点)」と「トレンド線(濃い線)」を重ねて表示

### 4.2 TDEE推定(週次、毎週月曜に再計算)
直近 `tdeeWindowDays`(21日)のうち、体重・摂取の両方が記録された日を対象とする。

```
avgIntake  = 対象期間の平均摂取kcal
ΔtrendKg   = trend(期間末) − trend(期間始)   // 減量中は負
periodDays = 期間の実日数

estimatedTdee = avgIntake − (ΔtrendKg × kcalPerKgFat / periodDays)
```

例:平均摂取1850kcal、21日でトレンド−0.9kg
→ TDEE = 1850 − (−0.9 × 7200 / 21) = 1850 + 309 ≈ 2159kcal

**ガード条件:**
- 記録日数が10日未満の場合は `initialTdee`(2324)または前回値を使用
- 推定TDEEの週次変動を±150kcalにクランプ(外食続き等での暴れ防止)
- 推定TDEEの絶対範囲を1600〜3200にクランプ

### 4.3 目標摂取カロリー算出
```
weeklyDeficitKcal = trendWeight × (weeklyLossRatePct / 100) × kcalPerKgFat
targetAvgKcal     = estimatedTdee − weeklyDeficitKcal / 7
targetByDay[d]    = round(targetAvgKcal × dayMultipliers[d] / 5) × 5
```

例:TDEE 2324、トレンド体重79kg
→ 週次赤字 = 79 × 0.0043 × 7200 ≈ 2446kcal
→ 平均目標 = 2324 − 349 ≈ 1975kcal
→ 平日 ≈ 1875kcal、土日 ≈ 2200kcal

**ガード条件:**
- targetAvgKcal の下限を基礎代謝相当(1500kcal)にクランプ
- 目標体重到達後は weeklyLossRatePct = 0(維持モード)へ自動切替を提案

### 4.4 PFC配分
```
proteinG   = round(trendWeight × proteinPerKg)          // 全曜日固定
proteinKcal = proteinG × 4
remainKcal = targetByDay[d] − proteinKcal
fatG       = round(remainKcal × fatRatio / 9)
carbG      = round((remainKcal − fatG × 9) / 4)
```

### 4.5 週次見直しフロー
1. 毎週月曜、アプリ起動時に前週分を自動計算
2. 「今週の目標」カードに新目標を表示、前週からの変更点(TDEE増減・目標kcal増減)を明示
3. ユーザーは手動で目標を上書き可能(オーバーライドフラグを保持)

```
┌─────────────────────────┐
│ 7/19(日)  高カロリー日      │
│                          │
│ 今日の目標               │
│ 2200 kcal               │
│ P 134g / F 69g / C 240g │
│                          │

│ │ [ 78.6 ] kg  [保存]  │ │
│ └─────────────────────┘ │
│ ┌─ 摂取記録(Claude算出値)┐│
│ │ kcal [    ] P [   ]  │ │
│ │ F  [    ]  C [   ]   │ │
│ │           [保存]      │ │
│ └─────────────────────┘ │
│ ※あすけんスクショをClaude │

```
- 「今週の残り」:週の目標合計から実績を引き、残日数で割った1日あたり可処分カロリーを表示(1日超過を週内で吸収する運用に対応)

### 5.3 「グラフ」タブ
- 体重チャート:実測(点)+ トレンド線(EMA)+ 目標到達予測線
- 期間切替:1ヶ月 / 3ヶ月 / 全期間
- TDEE推移チャート(週次棒グラフ)
- 統計サマリ:直近4週の平均減量ペース、目標到達予測日

### 5.4 「設定」タブ
- 目標体重・週次ペース・P/kg・脂質比率・曜日配分の編集
- JSONエクスポート(クリップボード + ファイルダウンロード)
- JSONインポート(貼り付け)
- 全データ削除(二段階確認)

### 5.5 UI要件
- 数値入力は `inputmode="decimal"` / `"numeric"` でテンキー表示
- 入力欄タップ時のズーム防止(font-size 16px以上)
- safe-area-inset対応(ノッチ・ホームバー)
- ダークモード対応(prefers-color-scheme)

## 7. テスト観点
- EMA計算:欠測日を挟んだ場合の連続性
- TDEE推定:記録10日未満・極端な摂取値(絶食日/爆食日)でのクランプ動作
- 週境界:月曜0時前後の入力が正しい週に属すること(ローカルタイムゾーン基準)
- localStorage上限・破損時のフォールバック(エクスポート済みJSONからの復元)
- iPhone Safari:ホーム画面追加後のlocalStorage永続性(※Safariは7日間未使用でデータ削除の可能性あり→PWAホーム画面追加で回避されるが、定期エクスポートを促すUIを入れる)

## 8. Claude Codeへの引き継ぎメモ
- リポジトリ構成:`index.html` 1ファイル + `icon-192.png` / `icon-512.png` + `sw.js`(Phase 3)
- 既存バックギャモンPWAのSafari最適化パターン(viewport設定、タップ遅延対策、apple-touch-icon)を流用すること
- コーディング規約:ES2020+、モジュールなし(単一ファイル内で IIFE または素のスコープ管理)、関数は純粋関数優先(アルゴリズム部はUI非依存でテスト可能に)
- アルゴリズム定数(α、クランプ幅、7200kcal/kg)はすべて `wt_config` 経由で変更可能にする
